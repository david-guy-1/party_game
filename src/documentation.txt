how to use :

git clone https://github.com/david-guy-1/game_engine.git
cd game_engine
npm i (takes a while)
npm run dev

----

Intro: Change the stuff in gamedata, do not change anything outside of that folder. 

gamedata/App is the first thing that gets rendered, and it should do the following in order : 
    add things to the "events" object from eventManager - see the EventManager section for how to do this. 
    create a gamedata object with a game. Add in the prop_fns in the app  (not gamedata, it should always say "prop_fns: {}" in GameData.ts). To see what prop functions are, look at the "for each render frame" section on prop_commands. This will require using clone_gamedata in case you want to re-use a scene. and globalStore (key-value pairs that will be passed into all functions)
    render ../GameDisplay.tsx. the props are data and globalStore

Declare the exact type of globalStore and events_type in gamedata/globalStore.d.ts. Also , set FPS here. 

----

A gamedata object defines the following functions  (see interfaces.d.ts for their args and return types):

game, draw_fn, anim_fn, sound_fn, init, button_click, prop_commands, prop_fns, reset_fn (not a function)
 

globalStore (key-value pairs - exact type declared in gamedata/globalStore.d.ts) is passed into all functions, so we can exchange data/state between them. - not that globalStore should only be used for the view, and not the game itself, to avoid coupling. The game object does not have access to globalStore. 

globalStore must have a key called "display" and value is display_type, see  the section on display_type for what it must contain.

GameDisplay first  calls init(g, globalStore) (from gamedata) in its useEffect ( equivalent to componentDidMount). It then calls set_events(g, globalStore), which tells the event listeners to listen to events.

the z-indices of GameDisplay elements are as follows:

for canvases : starts at 1, then goes upwards (1, 2, 3, ...) if "background" is in its name, subtract 1000.

images are at -1 

otherwise, they are rendered in the order : canvas, button, image, text

It then runs the following event loop: Every (some amount of time depending on computer speed), it calls g.tick until the number of game ticks matches the amount of time passed since last call. Thus, there can be multiple game ticks in a single render frame. But at least one game tick must pass between each render frame. 

The first thing that is called is init from gameData.

    for every game tick:

        call g.tick - no args. It returns a list of events. The lists for each game tick in a render frame are concatenated.

    for every render frame : 

        call the prop_commands(g, globalStore, events). See below for what it does. 

        call sound_fn(game, list of events) that outputs a pair [newsound : string | undefined, playsounds : string[]], first is the new BGM, second is a list of sound effects. - this only changes the BGM if it's different from the current one. If the first arg is undefined, the sound is muted. If the first arg is "no change", it's not changed

        for each canvas, calls draw_fn(game, event list, canvas name), which returns a pair (list of draw_command, should_clear) , and draws them on canvas  - if the second element is true, then the canvas is cleared, otherwise, it's not. IMPORTANT: if there is a background, it must be drawn on its own canvas at the very back. Otherwise the background might cover other things, such as animations.

        calls anim_fn(g, event list), which returns a list of animations. These animations are added to the list of ongoing animations.  then updates and draws them - calls each update and draw function with (g, globalStore).

When a button is clicked, button_click(game, globalStore, name of button) is called, it returns the same list as prop_commands: a list of functions to call, and arguments  

----

display type has the following , it's in interfaces.d.ts and the comments tell you what they mean. When GameDisplay is rendered, it reads globalStore.display and uses that. This means that to update it when it changes, you must re-render. See prop functions for how to do so. 

The "refs" contain a ref to the currently displayed canvas elements, so the functions in App or GameData can call functions that require canvas elements. 

----

Prop functions are functions that App.tsx adds to gameData that can be called when things happen in the game. They're stored in a key-value pair in gameData. The key is a string (its name) and the value is a function, which takes as args (game, globalStore, arg) (see below for what the last arg is). Every render frame, prop_commands(g, globalStore, events) is called, which returns a list of props functions that should be called, and arguments to call them with. 

Each element in the list returned by prop_commands is a pair (name : string, arg : any). If a function with that name exists, it's called with the second element in the pair as its last argument. 

There are special cases : if the first element is "rerender" , the GameDisplay object is re-rendered (which does NOT call init) , if it's "reset", reset_fn() (from gamedata) is called, then init is called, and the game is re-rendered

Thus, an easy way to change the display is to first  change globalStore.display_type, then call rerender. 


----


Animations are bound to games. They contain an update function (returns boolean, should it be removed?), a draw function (a list of draw commands), and optionally an end function (void). 

On each frame, update_and_draw is called, which does the following:

first, it calls each animation's update function, and if it returns true, calls  the end function (if it exists), then removes it from the list of animations (note the end function is called first, then it's removed).

After all animations are updated, it calls each animation's draw function, and draws it on the canvas specified by the animation object. 

----

EventManager handles all events except button clicks - it listens for mousemove, mousedown, mouseup, click, touchstart, touchend, keydown and keyup events.

It contains a list of events (should be set in App) , which is a key-value pair of type <string (name), eventCall>. An eventCall is a pair [function, params]. The function is called with args (event, game, globalStore, params), and it should not output anything. 

If the name contains the strings, the corresponding function will be called with the args specified in the previous paragraph: mousemove, mousedown, mouseup, click, touchstart, touchend, keydown,  keyup

Sound contains the functions : 

playSound(s : string)
toggleMute()
setMuted(x : boolean)
changeSound(s : string | undefined)
getCurrentTrack()
getMuted()

unlike sound_fn (from gameData), when changeSound is called directly, it always plays the given track from the beginning, even if it's already being played. 

---

img_with_center is a class that can be used to avoid drawing multiple shapes, and replace them with an image. 

construct it with the following parameters : commands, x, y, width, height. 

It draws the commands on a fresh canvas, takes the top-left width/height portion (so anything with negative x and y coords are gone).

Then , it sets the given x and y origin. 

then, call its "output" method with an x and y, it returns a drawImage_command. The class's x and y and the given x and y will line up. 

----

fade (defined in process_draws.ts) takes a list of draw commands, a canvas, and optionally a callback (arg : any, midpoint : boolean), arg to that callback , time (seconds),size,  and full (boolean). and draws a translucent rectangle of that color on the canvas repeatedly until half the time has passed, then clears the canvas, calls the callback with midpoint=true, draws the list of draw commands, and then draws a (ever more transparent) translucent rectangle. Finally, it calls the callback function with the specified args and midpoint = false. 

If full is false, then it just draws the rectangle, but fade out is instant. In this case, the time taken is half of the specified time, and the midpoint call does not happen.

tips : make a reset function that resets the game and globalStore, and put it in the init functions . 

use flags extensively - is an animation playing? is it fading? 

put a "props to run" in globalStore, and append to the output of prop_commands (and clear the globalstore) it every time prop_commands is called, so other commands can call prop functions 

put a blank "fade" layer at the top , and don't clear it when draw_fn is called. This is because draw_fn will draw on other layers and clear the fading. 

draw_fn : MUST check the layer carefully, don't draw the same thing on multiple layers.
